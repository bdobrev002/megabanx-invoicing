# MegaBanx v2 — Absolute Rules

This repository is the **clean rewrite** of the legacy `/opt/bginvoices` (v1) system.
v1 accumulated ~5000-line monolithic files plus `patch_*.py` scripts that mutate
`main.py` and the database in place. **None of that is allowed here.**

These rules are enforced mechanically by ESLint, a pre-commit hook, and CI.
Bypassing them (commits with `--no-verify`, direct pushes to the server, ad-hoc
`ALTER TABLE`, etc.) is a hard violation of the v2 contract.

---

## 1. Forbidden forever

### 1.1 Patch-style scripts
- No `patch_*.py`, `patch_*.js`, `*.patch`, `*.inject.js`, or any file whose
  purpose is to mutate another file at deploy time.
- Schema changes go through **Alembic migrations** (`backend-v2/alembic/versions/*.py`),
  committed to git and applied via `alembic upgrade head` in CI / deploy.
- Behavioural changes go through a normal PR against `main`.

### 1.2 DOM manipulation
- No `document.createElement`, `document.write`, `document.getElementById`,
  `document.querySelector`, `document.querySelectorAll`, or any other imperative
  DOM API.
- No `Element.innerHTML =`, `Element.outerHTML =`, `insertAdjacentHTML`.
- No `dangerouslySetInnerHTML` — use JSX or the React element tree.
- React components render via JSX; state lives in hooks / Zustand stores.

### 1.3 Dynamic code
- No `eval(...)`.
- No `new Function(...)`.
- No `setTimeout("string", ...)` / `setInterval("string", ...)` (implied eval).

### 1.4 Inline scripts in HTML
- `frontend-v2/index.html` is the ONLY HTML file. It contains a single
  `<script type="module" src="/src/main.tsx">` tag injected by Vite.
- No inline `<script>…</script>`, no inline `on*` attributes.
- No build-time script that edits `dist/index.html`.

### 1.5 Direct edits to build output
- `frontend-v2/dist/` is regenerated by `vite build`. **Never edited by hand,
  never committed to git, never patched on the server after deploy.**
- CI refuses any PR that stages files under `dist/`.

### 1.6 Direct production mutations
- No `ssh root@… "psql -c 'ALTER TABLE …'"`.
- No `scp file.js root@…:/opt/megabanx-v2/frontend-dist/`.
- Deploy is always: PR → merge `main` → CI artifact → deploy script.

### 1.7 Reusing v1 source
- `/opt/bginvoices/**` is **reference only** — read to learn what v1 does, never
  copy-paste. Its structure is the anti-pattern we are moving away from.
- When porting a feature, copy the **visual design** (via Playwright snapshot
  against `https://megabanx.com`) and the **business rules**, but re-implement
  the code idiomatically in the v2 modular layout.

## 2. Project layout (do not drift)

```
megabanx-invoicing/
├── .agents/RULES.md          ← this file
├── .github/workflows/        ← CI (lint, build, validate, tests)
├── .husky/                   ← pre-commit hooks
├── scripts/
│   ├── check-no-dist.js      ← blocks dist/ and patch files from commits
│   └── validate-build.sh     ← blocks unexpected dist artefacts
├── Architecture-2.0.md       ← source of truth for design
├── megabanx-2.0-plan.md      ← feature plan, updated as stages close
├── backend-v2/
│   ├── alembic/              ← schema migrations (the ONLY way to evolve the DB)
│   ├── app/
│   │   ├── main.py           ← FastAPI entry, no create_all in prod
│   │   ├── config.py
│   │   ├── database.py
│   │   ├── models/           ← one file per domain aggregate
│   │   ├── routers/          ← one file per REST resource
│   │   ├── schemas/          ← Pydantic request/response
│   │   ├── services/         ← business logic
│   │   └── utils/
│   └── pyproject.toml
└── frontend-v2/
    ├── eslint.config.js      ← enforces rules in §1.2–§1.4
    ├── index.html
    └── src/
        ├── api/              ← one file per REST resource
        ├── components/       ← small focused components
        ├── hooks/
        ├── pages/            ← route-level composition
        ├── stores/           ← Zustand
        └── types/
```

Never merge a PR that adds a file outside this tree without updating
`Architecture-2.0.md` in the same PR.

---

## 3. Workflow requirements

1. Every change lands via PR. No direct push to `main`.
2. Every PR passes CI (`lint`, `build`, `validate-build.sh`, `ruff`, tests).
3. Every schema change ships an Alembic revision in the same PR as the model change.
4. Every new endpoint is documented in `Architecture-2.0.md` §4 in the same PR.
5. Every new table/column is documented in `Architecture-2.0.md` §3 in the same PR.
6. Commits are signed/verified by GitHub; `--no-verify` is forbidden.

---

## 4. If a rule blocks you

- The rule is probably right. Re-read §1.
- If you believe the rule is wrong for a specific case, open an issue / PR that
  **amends this file**. Do not bypass it silently.
